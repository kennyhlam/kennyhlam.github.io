Getting Started

2016.08.06 · golang, golang-hugo, technology

So I haven’t created a website in quite a while, and well starting a blog has been on my backburner for quite a while. Anyways, I was basically inspired by a friend of mine’s blogging to get back into writing (thanks? Dennis). I’m hoping that I’ll branch back out again into things that aren’t technology-related, but this one will be. Let’s start at the beginning. I’m a back-end engineer, which basically means that anything I’ll be making may look…less than glamorous, but it’ll work however it might look.

To start out, I wanted to use GitHub Pages because it’s free (but what isn’t nowadays?) and because I’m a huge fan of GitHub. The alternative was Medium, and they got some flak in the past about their Terms of Service and basically being able to do a bunch of stuff without you knowing/having your explicit consent, but they’ve since gotten friendlier terms. Now, I’m not particularly worried about my blog getting re-produced, but it is nice to have the feeling that your output is your own, and Github has a great set of terms related to content ownership.

In any case, though, Medium has a very sexy look and feel to it, but the big issue is that I would only be working on writing; I wanted to work on something where I’d be learning something else, and what better way to learn things than to customize the heck out of your own blog! GitHub Pages is also super easy to use, where everything is just a git push -f away! There’s a whole bunch of other services which provide free/cheap hosting (Heroku, DigitalOcean, Amazon, etc. etc.), but there’s usually some amount of configuration and such to get it to hook into your repo, so why add in an extra middle-man when you can go directly to a repo itself.

The good people at Github recommend using a static website generator to help with the process, and they recommend Jekyll. Woe is me, there had to be another option! I write a bunch of Ruby code at work, and I can tell you that Ruby is awful. I mean it’s syntactic sugar, just like Python–you can basically write down some English and there’s a good chance it’ll run. The issue though, is that it’s slow, and it’s got bad support for almost-kind-of-maybe-sure-but-jokes-no “threads”.

There’s a few options out there, but I won’t list them all. In the end, I wanted something that wasn’t Ruby so I could learn something new, and I have a huge crush on golang, so Hugo was the obvious choice.

I followed the quickstart guide and quickly got started picking a theme, here are some of my favorites:

  • identity
  • material-design
  • strata
  • zen

I chose the minimalist theme for what you’re seeing now.

If you’re like me, you sure didn’t follow the guide to a tee. In the guide, though, they do say you can have sections of your website like blog, article or tutorial. However, you should be careful which template you’re using, since mine wasn’t rendering really anything cause I had run

hugo new blog/getting-started.md

Turns out some of the templates specifically look for keys like post; a variety of things are key’d off this, including the RSS feeds, so if you’re deviating from the norm, be careful to go over and find the references. If you’re like me, and you happen to be using a VM to do your development, you may also run into issues–you really want to bind to all addresses, 0.0.0.0. The last little tidbit is that you basically want to copy all the layouts and static from the themes into your own Hugo directory (I imagine this is how they meant for people to use this). It becomes really confusing to keep track of where exactly you are in the directory when there’s a nested theme that has the same directory structure, so copying them into your top-level Hugo directory overrides the theme’s defaults.

At the end of the day, what you really want is probably (note http for local development)

hugo server --buildDrafts --bind 0.0.0.0 --baseUrl=http://<YOUR IP>/ 2>&1 > /dev/null &

for Hugo to silently run in the background with it’s auto-update feature.

When you finally finish, careful that in your config.toml, you have https if you’re using Github Pages. At least for Chrome, you’ll get mixed content messages (meaning some assets are https and some are http), but since your Github Pages url is https, you’ll be missing resources and your website will look like it hasn’t loaded its css (since it hasn’t).

Final thoughts

Hugo seems like a rather opinionated framework for static website generation, which isn’t necessarily a bad thing. However, it’s relatively young, and that’s always a bit risky. I haven’t finished reading the docs (much more of a do-er than a read-er), and some things do feel a bit clunky (you can only hugo in the top level where your config.toml is, when you publish old assets aren’t removed, there’s a lack of helpful feedback such as when I made blog instead of posts, etc.) A lot of things feel like they’ve been developed so far, but not enough to the point where the defaults suit the typical use-case.

All that said, though, it was relatively easy to set up and get started with Hugo, and it provides a rather intuitive way to build a static website. While some things can be finessed, such as how to deal with your public directory and nested git submodules, it looks clean at the end with just some initial startup time, but writing itself was rather streamlined. I haven’t tried the other options for website generation, but I’m excited at the prospect of Hugo and where it can go in the future.
